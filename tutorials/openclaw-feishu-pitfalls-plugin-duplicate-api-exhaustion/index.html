OpenClaw 飞书部署避坑：重复插件冲突与 API 调用耗尽问题全解

摘要：本文针对 OpenClaw 飞书插件部署中的两大高频问题——重复插件 ID 冲突和API 调用额度快速耗尽，从成因分析、排查方法到根治方案进行完整梳理，提供可直接执行的命令与配置代码，帮助你快速打通部署链路。

问题一：网关启动提示「duplicate plugin id detected: feishu」

在 OpenClaw 升级或网关启动过程中，可能会出现以下警告：

Config warnings:
- plugins.entries.feishu: plugin feishu: duplicate plugin id detected; 
  later plugin may be overridden 
  (/usr/lib/node_modules/openclaw/extensions/feishu/index.ts)

仔细检查了配置文件里的 plugins 部分，并没有看到重复的 feishu 插件条目。这说明「duplicate plugin id detected」的警告并非来自本地配置文件，而是 OpenClaw 升级过程中内部加载插件时产生的冲突。

问题根源分析

虽然配置文件里只显式配置了一个 feishu 插件，但 OpenClaw 在升级时会同时加载：

  1. 配置文件中手动启用的 feishu 插件（~/.openclaw/openclaw.json 里的配置）
  2. OpenClaw 内置的 feishu 插件（系统路径 /usr/lib/node_modules/openclaw/extensions/feishu/）

关键变化：2026.2.x 版本后，OpenClaw 官方已将飞书插件（@openclaw/feishu）内置到核心包中，安装路径为 /usr/lib/node_modules/openclaw/extensions/feishu。

而老版本用户手动安装的飞书插件仍残留于用户扩展目录 /root/.openclaw/extensions/feishu，两个插件 ID 均为 feishu，加载时触发冲突检测。

解决方案（无需改配置，直接修复加载冲突）

核心思路：直接删除用户目录下的旧版飞书插件，保留官方内置版本。

⚠️ 重要提示：早期解决方案是"保留自己装的、删除系统的"，但升级后系统会重新安装内置插件，导致问题复发。因此推荐删除用户自装版本。

正确的删除操作步骤（避免踩坑）

1. 先停止 OpenClaw 相关进程（可选但推荐） 

# 终止可能正在运行的 OpenClaw 进程
pkill -f openclaw

2. 安全删除内置 feishu 插件目录 

# 直接删除冲突的内置插件目录
rm -rf /root/.openclaw/extensions/feishu

# 验证是否删除成功
ls /root/.openclaw/extensions/feishu | grep feishu
# 执行后如果没有输出，说明删除成功

3. 关键：重新加载插件并验证

删除后必须执行这一步，让 OpenClaw 重新识别插件配置，避免插件失效：

# 重新加载插件
openclaw plugins enable feishu

# 运行健康检查，自动修复插件依赖/加载问题
openclaw doctor

注意事项

✅ 如果手工删除飞书目录后，你发现在配置文件 /root/.openclaw/openclaw.json 里面的 install 部分还能找到飞书的内容：

有两种处理方式：

  1. 手动删除：可以删除整块内容，但注意大括号别删多了导致 JSON 格式出错
  2. 命令卸载：直接在终端输入以下命令：
openclaw plugins uninstall feishu

问题二：飞书 API 调用很快耗尽

有没有遇到这样的情况：好不容易安装了 OpenClaw，折腾半天终于接入了飞书，结果还没聊几下，飞书提醒你 API 额度已使用 90%，第二天直接耗尽…

问题本质

默认情况下，OpenClaw 的飞书插件每分钟都会发起健康检查。

算笔账：

1 分钟 1 次 × 60 分钟 × 24 小时 × 30 天 = 43,200 次/月

飞书免费版额度才 50,000 次/月，这就没了？

查询 API 使用情况

首先，教大家怎么查询 API 调用的日志和使用情况：

  1. 登录 飞书开放平台 → 飞书管理后台

  2. 右上角可以看到使用量及额度（现在额度提升到了 100 万）

  3. 在左侧菜单找到 合规 → 日志审计

  4. 点击 OpenAPI 日志

  5. 选择时间，点击查询

  6. 看到大量的 undefined / GET/open-apis/bot/v3/info，这就是罪魁祸首！

这货不停地在消耗你的额度，我们的目的就是把它干掉。

解决方案：添加 24 小时缓存

核心思路：给 probe.ts 加上 24 小时内存缓存。

来源：https://xx0a.com/blog/openclaw-feishu

文件位置：/usr/lib/node_modules/openclaw/extensions/feishu/src/probe.ts

// 在文件顶部添加缓存常量
const PROBE_CACHE_TTL_MS = 24 * 60 * 60 * 1000; // 24 小时
const probeCache = new Map<string, { result: any; timestamp: number }>();

// 在 probe 函数中，先检查缓存
const cacheKey = `${gatewayUrl}:${port}`;
const cached = probeCache.get(cacheKey);
if (cached && Date.now() - cached.timestamp < PROBE_CACHE_TTL_MS) {
  return cached.result;
}

// 正常发起请求...
const result = await fetch(...);

// 写入缓存
probeCache.set(cacheKey, { result, timestamp: Date.now() });
return result;

⚠️ 注意事项：因为版本更新，不要全部照搬原文代码。推荐的方法是把本文链接或原文链接发给 OpenClaw，让它自己修改，它会自动适配当前版本。

优化效果

API 调用从 43,200 次/月 降到约 30 次/月（每分钟 1 次变每天 1 次），节省 99.9%。

优化完成后，基本就看不到 undefined / GET/open-apis/bot/v3/info 了，一般只在网关重启时会出现几次。

总结

  1. ✅ 直接删除 /root/.openclaw/extensions/feishu 是可行且有效的，能解决插件 ID 重复问题
  2. ✅ 删除后执行 openclaw doctor 重新检查插件
  3. ✅ 最简单的方案：在最初安装 OpenClaw 时直接跳过 channel 那部分，很多人就是按照旧教程重复安装飞书导致问题
  4. ✅ 给飞书调用加上缓存可以大幅降低 API 调用，可以愉快地白嫖飞书了
