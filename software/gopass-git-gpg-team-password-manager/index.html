gopass：把团队密码交给 Git 和 GPG，是复古还是清醒？

密码管理器这件事，最尴尬的地方从来不是“怎么存一个密码”。

真正麻烦的是：一群人怎么共享一堆秘密，同时别把秘密变成一团没人敢碰的黑箱。

账号密码、API Token、服务器证书、数据库连接串、临时 SSH 凭据、第三方平台回调密钥……这些东西像厨房里的刀。每个人都要用，但没人希望它们躺在微信群、飞书聊天记录、Excel 附件、Notion 页面、开发机桌面上的 密码.txt 里。

SaaS 密码管理器当然能解决一部分问题。1Password、Bitwarden、Dashlane 这类工具把 UI、浏览器插件、团队权限、审计日志都包好了。对大多数公司来说，这是最省心的路线。

但 gopass 走的是另一条路。

它不试图把密码管理变成一座云端宫殿，而是把它拆回三个开发者已经熟得不能再熟的积木：

  • 文件：每个 secret 是一个文件
  • GPG / age：每个文件独立加密
  • Git：变更历史、同步、分支、冲突处理都交给 Git

听起来像上世纪 UNIX 哲学留下来的化石。但有趣的是，到了 AI Agent、自托管服务、CI/CD 密钥满天飞的 2026 年，这种“复古”反而显得有点清醒。

gopass 不是又一个漂亮密码 App

gopass 官方给自己的定位很直接：

The slightly more awesome standard unix password manager for teams.

它是经典 UNIX 密码管理器 pass 的 Go 语言重写和增强版。默认情况下，gopass 使用 GPG 加密凭据，用 Git 给密码仓库做版本管理。它主要面向命令行用户，也可以通过 gopassbridge 接入 Firefox、Chrome 等浏览器。

官方文档里有几句很关键：

  • 它是 pass 的 drop-in replacement
  • 默认用 gpg encryption 和 git storage
  • 可以改用 age 等加密后端
  • 可以不用 Git，改成普通文件系统存储
  • 主界面是命令行，适合 CLI 用户、CI/CD 系统和自动化流程
  • 支持团队共享、多个 password store、recipient 管理、浏览器集成、Git credential helper、HIBP 泄露检查等扩展

换成人话：gopass 不是给“我忘了 Netflix 密码”的人准备的，它更像给开发团队和基础设施管理员准备的一把折叠刀。

它不怕你打开终端。它甚至默认假设你愿意打开终端。

这点非常重要。因为密码管理器一旦进入团队协作场景，就不只是“自动填表”了。它会碰到很多更脏、更工程化的问题：

  • 密钥怎么进 CI？
  • 谁有权限解密某个服务的 secret？
  • 离职成员怎么移除？
  • 密码更新后怎么留下可追溯历史？
  • 多台开发机、跳板机、离线机器怎么同步？
  • 密钥仓库能不能放进公司已有 Git 基建？

传统密码 App 往往回答：“来我的平台里配置。”

gopass 的回答是：“你已经有 Git 和 GPG 了，为什么不把它们用起来？”

它的核心设计：密码不是数据库记录，而是加密文件

很多密码管理器会把所有 secret 放进一个数据库，再用主密码或密钥派生机制整体保护。gopass 的模型更接近 pass：每个 secret 是一个单独文件。

官方文档里的示例很直观：

.password-store
└── laptops
    ├── dell.gpg
    ├── hp.gpg
    └── macbook.gpg

在 gopass 里看，可能是这样：

gopass
└── laptops
    ├── dell
    ├── hp
    └── macbook

也就是说，laptops/macbook 这个 secret，对应底层一个 macbook.gpg 加密文件。

这套结构有两个好处。

第一，它很容易理解。没有神秘数据库，没有专有格式，没有必须依赖某个云平台才能打开的保险柜。你知道 secret 在哪里，也知道它怎么被保护。

第二，它天然适合 Git。新增一个密码，就是新增一个加密文件；修改一个密码，就是修改一个加密文件；删除一个密码，就是 Git 里删一个文件。历史记录、commit message、冲突、远端同步，全是 Git 擅长的东西。

当然，代价也摆在桌面上：文件名和目录结构可能暴露元数据。

比如你把 secret 命名为：

production/mysql/root-password
aws/company-main/root-access-key
stripe/live/webhook-secret

即使文件内容是 GPG 加密的，别人只要拿到仓库，也能从路径里看出你有哪些系统、哪些服务、哪些权限边界。这不是 gopass 独有的问题，pass 生态也一样。

所以 gopass 官方文档也提醒：不要把敏感信息放进 secret 名称里。

这句话看起来不起眼，实际是使用 gopass 的第一道分水岭。会用的人会把路径设计成合理索引；不会用的人会把自己的基础设施拓扑图写进文件名。

Git 是同步工具，也是审计工具

gopass 最有意思的点，是它把 Git 用到了一个非常自然的位置。

在密码管理场景里，同步和审计通常是 SaaS 产品的卖点：谁什么时候改了哪个密码，哪些设备同步了，哪个版本被恢复了。

而 gopass 说：这些事情 Git 本来就在做。

一个 password store 可以是一个 Git 仓库。你可以 clone：

gopass clone git@example.com/pass.git

也可以把工作仓库挂成一个单独 store：

gopass clone git@example.com/pass-work.git work

然后通过：

gopass sync

同步远端仓库和 GPG keys。

这就让团队密码仓库变成了一个很“工程”的东西：

  • Git remote 可以是 GitLab、Gitea、GitHub Enterprise、自建 SSH Git 仓库
  • 权限可以叠加 Git 仓库访问控制和 GPG recipient 解密控制
  • commit history 记录变更轨迹
  • 冲突可以用 Git 的方式处理
  • 备份可以沿用已有 Git 备份体系

它不一定比 1Password 的管理后台舒服，但它非常透明。

透明的价值，在小团队、基础设施团队、自托管团队里会被放大。因为这些团队往往已经有 Git 工作流，也习惯用 code review、SSH key、GPG key、CI pipeline 管基础设施。

密码仓库如果也能进入这套系统，就少了一套独立心智负担。

但也别浪漫化 Git。

Git 管密码仓库，意味着你必须真的懂 Git。一次错误 push、一次仓库权限配置失误、一次把仓库设成 public 的手滑，都可能比普通项目代码泄漏更可怕。

gopass 的安全边界不是“仓库私有就安全”，而是“仓库就算被拿走，内容仍然由加密保护”。这就是 GPG recipient 的意义。

GPG recipient：团队共享的钥匙圈

单人用密码管理器，核心是“我能解密”。

团队用密码管理器，核心变成：“谁能解密，谁不能解密，以及这个名单怎么变。”

gopass 比原始 pass 更强调 recipient management。官方 features 文档把它列为 stable 功能：可以管理每个 store 的多个用户，也可以为不同 store 配置不同 recipient。

一个团队 store 可以有多名成员的 GPG 公钥。secret 在写入或更新时，会加密给这些 recipient。只有拥有对应私钥的人才能解密。

这套模型很适合团队分层：

  • personal：个人密码
  • work：团队共享服务
  • prod：生产环境密钥
  • client-a：某个客户项目
  • ci：流水线专用凭据

gopass 支持 multiple stores，可以像文件系统一样把多个 store mount 起来。这样每个 store 都能有自己的 recipient 列表、Git remote、同步策略。

这比“一整个公司所有 secret 塞进一个大保险柜”健康得多。

不过，GPG 也不是魔法。它有自己的难点：

  • 新成员需要生成、交换、信任公钥
  • 老成员离开后，需要移除 recipient，并重新加密相关 secret
  • 私钥丢失会导致无法解密
  • GPG 本身对新人不友好
  • YubiKey、智能卡、agent、pinentry 等生态细节会吓退很多普通用户

所以 gopass 很适合“开发者团队”，但不一定适合“全公司所有员工”。财务、人事、运营如果只是想安全存一个账号，1Password 或 Bitwarden 的体验大概率更好。

gopass 的价值不是取代所有密码管理器，而是让技术团队把 secret 管理纳入自己熟悉的工具链。

它比 pass 多出来的东西，才是团队可用的关键

如果只看“GPG 加密文件 + Git 同步”，经典 pass 已经能做。gopass 真正的意义在于：它把 pass 的极简模型向团队和现代工作流扩了一圈。

官方 features 文档列了不少增强项：

  • Multiple stores：多个密码仓库挂载
  • Recipient management：管理不同 store 的用户
  • Password quality assistance：离线检查新旧密码弱点
  • Password leak checker：通过 gopass-hibp 做离线泄露检查
  • JSON API：给浏览器插件等集成使用
  • Automatic fuzzy search：找不到精确条目时自动模糊搜索
  • gopass sync：同步远端仓库和 GPG keys
  • Desktop Notifications：长任务完成时桌面通知
  • REPL：直接运行 gopass 进入带补全的交互 shell
  • OTP support：支持 TOTP / HOTP
  • Extensions：通过 API 扩展自定义命令

这些功能看起来零散，放进真实工作流里就很顺。

比如开发者拿 GitHub token：可以用 git-credential-gopass。浏览器登录后台：可以用 gopassbridge。Kubernetes secret：有 kubectl-gopass。Terraform 工作流：也有 provider 可以对接。

这就是 gopass 的优势：它不是一个孤岛，而是把自己藏进开发者每天已经在用的缝隙里。

SaaS 密码管理器的哲学是“来我的界面”。

gopass 的哲学是“我去你的工作流里”。

CI/CD 场景里，它比图形密码管理器更自然

很多人讨论密码管理器时，只想到浏览器自动填充。但开发团队最常见的 secret 消费者，未必是人，可能是脚本。

CI/CD pipeline 要拿部署 token。

本地脚本要读 API key。

Terraform 要读云平台凭据。

备份任务要拿 S3 access key。

这些场景里，图形密码管理器往往会变笨。它们适合“人点一下”，不一定适合“程序可靠地拿到某个 secret”。

gopass 的命令行接口反而简单粗暴：

gopass show -c foo

复制到剪贴板。

gopass show service/api-token

输出 secret。

gopass generate work/aws-access-key 24

生成一个新 secret。

当然，CI 里直接输出 secret 是危险操作，实际使用要结合权限、runner 环境、日志屏蔽、短期凭据和最小权限原则。但至少从工具形态上，gopass 更接近“基础设施的一部分”，而不是“桌面软件的一部分”。

这也是它吸引开发者的地方：你可以把 secret 管理写进脚本，但 secret 本身仍然由 GPG/age 和 Git 保护。

gopass 的缺点：安全感来自理解，不来自包装

gopass 最大的问题，不是功能不够，而是它不给你太多“保姆式安全感”。

SaaS 密码管理器会把复杂度藏起来。你创建团队、邀请成员、设置 vault、点权限，剩下的由平台处理。你当然还是要信任平台，但体验更直观。

gopass 则把复杂度摊在桌面上：

  • Git remote 放哪？
  • SSH key 怎么管？
  • GPG key 怎么备份？
  • recipient 怎么轮换？
  • 文件名怎么设计？
  • secret 更新怎么通知团队？
  • 离职成员移除后旧密钥是否需要轮换？
  • 多人同时改同一个 store，冲突怎么处理？

它不会替你做组织治理。

它只提供一套很透明的技术底座。

这意味着，gopass 的门槛不是命令行，而是安全意识。你必须知道：

  • Git 仓库泄漏仍然可能暴露元数据
  • GPG 私钥丢失不是小事
  • 删除 recipient 不等于旧 secret 自动失效
  • 共享密码永远不如分发独立账号和最小权限 token
  • TOTP secret 和密码放一起会降低双因素意义

官方文档在 OTP 部分也提醒：是否应该把 OTP secret 和密码放在一起，需要按安全需求判断，理想情况下可以用 multiple stores 隔离，或者用需要用户交互的硬件 token。

这类提醒很重要。因为 gopass 太灵活，灵活到可以被用得很好，也可以被用得很糟。

谁适合用 gopass？谁不适合？

适合 gopass 的团队，通常有几个特征：

  • 成员以开发者、运维、SRE、DevOps 为主
  • 已经使用 Git 和 SSH 工作流
  • 能接受 GPG 或 age 的密钥管理成本
  • 重视自托管、离线可用、透明格式
  • 需要把 secret 接入 CLI、脚本、CI/CD 或基础设施工具
  • 不想把所有团队密码完全托管给第三方 SaaS

不适合 gopass 的场景也很明确：

  • 团队成员不熟悉命令行
  • 主要需求是浏览器自动填表和移动端体验
  • 管理者需要现成后台、细粒度审计、合规报表
  • 公司没有人愿意负责密钥轮换和仓库权限
  • 只想“买个工具，马上全员用起来”

如果你的团队是第二类，别硬上 gopass。工具不是信仰，密码管理更不是展示技术洁癖的地方。

但如果你的团队本来就在自托管 GitLab、写 Ansible、跑 Terraform、用 YubiKey、维护一堆内部系统，gopass 会显得非常自然。

它像一把机械键盘：普通人觉得麻烦，重度用户觉得每个键程都可控。

和 1Password、Bitwarden、pass 怎么看待差异？

粗略比较，可以这么理解：

1Password 更像成熟商业产品。体验、团队管理、浏览器插件、移动端、企业能力都强，适合不想折腾的团队。代价是订阅费用和平台依赖。

Bitwarden 更像开源与商业之间的平衡点。它有云服务，也能自托管，UI 体验比 gopass 友好，团队采用门槛低。对多数中小团队，Bitwarden 可能是最实际选择。

pass 是 UNIX 哲学原版：极简、透明、强依赖 GPG 和文件系统。它漂亮，但团队增强能力有限，需要靠扩展和约定补齐。

gopass 则站在 pass 和团队工程之间。它保留了“加密文件 + Git”的透明结构，又补了 multiple stores、recipient management、sync、JSON API、OTP、浏览器和开发工具集成。

所以 gopass 不是“更便宜的 1Password”。这个理解错了。

它是“更适合开发者工作流的 pass 增强版”。

最值得写进团队规范的，不是安装命令

如果团队真要用 gopass，最重要的不是这几条命令：

gopass setup
gopass init
gopass create
gopass ls
gopass show -c foo
gopass sync

这些命令很快就能学会。

真正该提前写进规范的，是这些问题：

  • password store 放在哪个 Git remote？
  • store 如何按环境、项目、客户拆分？
  • secret 命名规则是什么？哪些信息禁止出现在路径里？
  • 新成员加入时如何添加 recipient？
  • 成员离开时如何移除 recipient、重新加密、轮换关键 secret？
  • GPG 私钥如何备份？是否使用硬件密钥？
  • CI/CD 读取 secret 的边界在哪里？
  • 是否允许把 TOTP secret 放进同一个 store？
  • 谁负责定期做泄露检查和弱密码检查？

没有这些规范，gopass 会从“开发者密码保险柜”变成“加密版共享文件夹”。

加密版共享文件夹听起来安全，实际只是把混乱包了一层 GPG。

结论：gopass 复古，但不是落后

gopass 最打动我的地方，是它没有假装密码管理可以靠一个漂亮界面彻底解决。

它承认 secret 管理本质上就是一件脏活：涉及人、权限、历史、设备、脚本、自动化、离职、泄漏、轮换。然后它把这件脏活交给开发者最熟悉的三个工具：文件、加密、Git。

这当然不适合所有人。

但对技术团队来说，这种设计有一种罕见的诚实感：

  • 不把数据锁进专有格式
  • 不强迫你依赖云服务
  • 不回避密钥管理复杂度
  • 不伪装成“点几下就安全”
  • 不把开发者工作流当二等公民

它像一只朴素但可靠的金属工具箱。没有香氛，没有丝绒内衬，没有“企业级安全体验”宣传片。但你打开它，里面是 Git、GPG、文件系统和一堆可以接进真实工程流程的接口。

如果你的团队要的是省心，选 1Password 或 Bitwarden。

如果你的团队要的是可控、透明、离线、可脚本化，并且愿意为此承担 Git/GPG 的学习成本，gopass 值得认真看一眼。

复古不等于落后。

有时候，复古只是把复杂度放回它该在的地方：交给懂它的人，而不是藏进一个你看不见的云端黑盒里。
